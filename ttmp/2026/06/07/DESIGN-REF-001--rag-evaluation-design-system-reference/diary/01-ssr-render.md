## Step 2: Fix CSS Module Extraction and Button Rendering

The first SSR output looked superficially complete, but the rendered button examples were not actually styled as design-system buttons. The HTML contained the scoped class names generated by Vite SSR, but the extracted CSS bundle did not contain the matching Button CSS Module rules.

### Prompt Context

**User prompt (verbatim):** "We're taking over for your little brother who has been losing the sauce. Update the diary, commit what you think should be (ttmp stuff should). Then make things work properly to make a proper design system HTML. You can also use the read() call to look at images, not like your little brother who is actually blind."

**Assistant interpretation:** Resume the ticket cleanly, document the broken path, fix the static HTML so it renders actual design-system styling, verify visually, and commit source plus ticket artifacts.

**Inferred user intent:** The user wants a reliable, handoff-quality design-system reference, not a half-styled HTML dump.

**Commit (code):** pending — after this diary update and final verification

### What I did
- Re-ran the SSR reference generator.
- Served the generated HTML from the ticket output directory in a tmux session:
  - `tmux new-session -d -s dsref 'python3 -m http.server 8911'`
- Used Playwright to inspect computed styles for `[data-rag-atom="Button"]`.
- Confirmed the broken state: buttons had `background: rgba(0,0,0,0)`, `border: 0px`, `padding: 0px`, and browser default `appearance: button`.
- Traced the cause to missing CSS-module output: the SSR HTML used `_root_1gwp3_1`, but the generated `<style>` block did not contain `._root_1gwp3_1{appearance:none...}`.
- Reworked `web/src/reference-pages/css-entry.ts` so CSS Modules are imported as bindings and assigned to `globalThis.__ragDesignReferenceCssModules`. This runtime side effect prevents Rollup from tree-shaking all module CSS out of the CSS extraction build.
- Rebuilt the references and verified the fixed computed styles:
  - default button: white background, black text, 1px black border, `2px 12px` padding, Monaco 11px metadata font
  - primary button: black background, white text, 1px black border
  - compact buttons: `1px 6px` padding
- Captured screenshots and read them through the image reader:
  - `foundation-fixed-preview.png`
  - `widgets-fixed-preview.png`

### Why
- Rendering React HTML is not enough; CSS Modules must survive the extraction pass with class names matching SSR output.
- This is the same failure class described in the Goja UI DSL article: real components look weak or broken when the standalone visual shell and token/CSS contract are incomplete.

### What worked
- Vite SSR produced deterministic CSS-module class names.
- Vite client build produced matching CSS once module CSS was forced to remain live.
- Playwright computed-style inspection caught the concrete defect precisely.
- `read()` on the screenshots confirmed the visual output: button variants now render as actual bordered/filled buttons; panels, tables, form controls, and composition examples render with the intended monochrome visual language.

### What didn't work
- Importing CSS Modules only for side effects in `css-entry.ts` was not enough. Rollup produced a zero-length JS chunk and only emitted `index.css` plus a partial subset of layout/foundation CSS.
- Direct CSS module imports without using the bindings were still shaken away.
- Temporary background HTTP servers started directly from shell invocations were unreliable; using tmux is the right pattern for longer browser inspection sessions.

### What I learned
- For this repository's Vite/Rollup setup, CSS Modules used only by SSR-rendered components need an explicit live binding in a client build entry if we want to extract their CSS into a standalone HTML file.
- The quickest diagnostic is computed style, not screenshot guessing:
  - bad button: `border: 0px`, `padding: 0px`, transparent background
  - good button: `border: 1px solid rgb(0,0,0)`, expected padding, expected background

### What was tricky to build
- The SSR and client-CSS passes are separate. SSR resolves the component class map and writes class names into the HTML; the CSS extraction build must independently import the same CSS modules and emit the same hashes. If Rollup tree-shakes those imports, the HTML looks structurally correct but visually wrong.
- The symptom was misleading because many foundation/layout styles did render, while atom styles (especially Button) were missing.

### What warrants a second pair of eyes
- Whether the CSS extraction entry should include every CSS module in `src/components/**/*.module.css` or only the modules used by the two reference pages.
- Whether the reference pages should use the app's `AppShell` chrome at the top level instead of a custom `ds-ref-page` wrapper.

### What should be done in the future

- Add a small smoke assertion in the script that fails if computed/static CSS does not include Button CSS (`appearance:none`, `._root_...` rules) before writing output.
- Add a WidgetRenderer IR page as a third reference sheet.

## Step 3: Typecheck and css-visual-diff Evidence

After the CSS extraction fix, I ran the frontend typecheck and found that several reference-page examples were visually rendering but not typed against the real component APIs. I fixed those so the examples use the same public props as application code.

### Prompt Context

**User prompt (verbatim):** (same as Step 2)

**Assistant interpretation:** Finish the reference generator as production-quality code: typed, validated, and documented.

**Inferred user intent:** Avoid another visually plausible but technically wrong artifact.

**Commit (code):** pending — to be committed after this diary update

### What I did
- Ran `pnpm typecheck` in `web/`.
- Fixed reference-page API mismatches:
  - `SelectInput` uses `<option>` children instead of an unsupported `options` prop.
  - `CheckboxRow` uses `checked` + `onChange` + children instead of an unsupported `label` prop.
  - `TabList` receives required `onChange`.
  - `AppNav` receives required `onItemSelect` and uses real `AppNavItem` shape.
  - Removed unused imports (`React`, `ScrollRegion`, `CodeText`, broken `IconButton` import).
- Re-ran `pnpm typecheck` successfully.
- Re-ran `node scripts/01-render-design-reference.cjs` successfully.
- Added `scripts/02-cssvd-inspect-reference.js`, a css-visual-diff verb that inspects Button, Panel, and DataTable computed CSS from a generated reference page.
- Ran:
  - `css-visual-diff verbs --repository ttmp/.../scripts design-ref inspect-reference http://127.0.0.1:8911/widgets-and-composition.html /tmp/design-ref-cssvd --output json`
- The css-visual-diff run returned `ok: true` with 3 inspected regions.

### What worked
- Typecheck is now clean.
- css-visual-diff confirms the generated page exposes stable selectors and inspectable computed CSS for the design-system primitives.
- The image reader confirmed the Widgets & Composition page now shows real bordered buttons, black primary buttons, panels, tables, form controls, and full composition patterns.

### What didn't work
- The previous reference page relied on invalid props that React ignored or tolerated. This is exactly the kind of drift that can make static reference pages lie about the component API.

### What I learned
- A visual reference must validate both visually and at the TypeScript API level.
- css-visual-diff is useful here as a regression aid because it gives concrete computed CSS evidence for key selectors, not just screenshots.

### What was tricky to build
- The React examples needed to be honest examples of the real component APIs. Some components expose simpler public APIs than the earlier examples assumed (`SelectInput`, `CheckboxRow`, `AppNav`, `TabList`).
