# Design System Unification Analysis and Implementation Guide

## Executive Summary

`web/src` and `packages/rag-evaluation-site/src` currently duplicate the same RAG design-system primitives. A machine-generated inventory found 74 shared component files; 72 are byte-for-byte identical. The package has no package-only component files. It is already a reusable subset of `web`, but `web` still imports its local copies.

The target architecture is:

```text
packages/rag-evaluation-site
  owns reusable design-system components, Widget IR types, WidgetRenderer, hooks, theme/css

web
  owns app routes, backend-connected containers, API services, store, and domain pages that are not yet package-stable
```

The migration should be incremental and visual-parity-driven. First, make `web` depend on the local package for development. Then redirect imports for shared primitives/layout/molecules/widgets. Only after typecheck, Storybook, and css-visual-diff parity pass should duplicated `web/src` files be deleted.

## Current-State Evidence

Different shared files:

The differing widget files are expected: the package versions include newer fixes and should become canonical.

## Development Dependency Model

Add a root pnpm workspace so `web` can depend on the local package:

```yaml
# pnpm-workspace.yaml
packages:
  - web
  - packages/rag-evaluation-site
```

Then add a workspace dependency to `web/package.json`:

```json
{
  "dependencies": {
    "@go-go-golems/rag-evaluation-site": "workspace:*"
  }
}
```

For source-level development, Vite and TypeScript should resolve package imports to source files rather than stale built artifacts:

```ts
// web/vite.config.ts
resolve: {
  alias: {
    '@go-go-golems/rag-evaluation-site': fileURLToPath(new URL('../packages/rag-evaluation-site/src/index.ts', import.meta.url)),
    '@go-go-golems/rag-evaluation-site/app': fileURLToPath(new URL('../packages/rag-evaluation-site/src/app/index.ts', import.meta.url)),
    '@go-go-golems/rag-evaluation-site/ir': fileURLToPath(new URL('../packages/rag-evaluation-site/src/widgets/ir.ts', import.meta.url)),
  },
}
```

```json
// web/tsconfig.json
{
  "compilerOptions": {
    "paths": {
      "@go-go-golems/rag-evaluation-site": ["../packages/rag-evaluation-site/src/index.ts"],
      "@go-go-golems/rag-evaluation-site/app": ["../packages/rag-evaluation-site/src/app/index.ts"],
      "@go-go-golems/rag-evaluation-site/ir": ["../packages/rag-evaluation-site/src/widgets/ir.ts"]
    }
  }
}
```

This gives `web` live package-source imports during development while the package remains buildable for distribution.

## Import Migration Strategy

### Before

```ts
import { Button } from './components/atoms';
import { Panel } from './components/layout';
import { DataTable } from './components/molecules';
import { WidgetRenderer } from './widgets';
```

### After

```ts
import { Button, Panel, DataTable, WidgetRenderer } from '@go-go-golems/rag-evaluation-site';
```

For feature components under `web/src/components/...`, relative depth varies, but the target package import stays stable.

Migration should start with only the byte-identical layers:

1. foundation
2. atoms
3. layout
4. shared molecules: `AppNav`, `DataTable`, `MetadataGrid`
5. widgets: `ir`, `cellRenderers`, `WidgetRenderer`, `actions`

Do not move web-only organisms/pages until their prop/API boundary is reviewed.

## Storybook and Visual Parity Strategy

The user requested Storybook-based visual parity and css-visual-diff. The intended validation loop is:

```text
1. Build or run Storybook for baseline web components.
2. Redirect imports to package components.
3. Build/run Storybook again.
4. Use css-visual-diff to compare the same Storybook story URLs/selectors.
5. Inspect review artifacts before deleting duplicated files.
```

Initial parity candidates:

- `atoms/Button`
- `atoms/TextInput`
- `foundation/Caption`
- `layout/Panel`
- `layout/Stack`
- `layout/TabList`
- `molecules/AppNav`
- `molecules/DataTable`
- `molecules/MetadataGrid`
- `pages/DslPreviewPage` or a fixture that exercises `WidgetRenderer`

Potential css-visual-diff direct comparison command:

```bash
css-visual-diff compare \
  --url1 http://127.0.0.1:6006/iframe.html?id=atoms-button--all-states \
  --selector1 '#storybook-root' \
  --url2 http://127.0.0.1:6007/iframe.html?id=atoms-button--all-states \
  --selector2 '#storybook-root' \
  --viewport-w 1280 \
  --viewport-h 900 \
  --threshold 30 \
  --out ttmp/.../sources/visual-parity/button
```

A project-local JS verb can later automate a story list and emit review-site data.

## Implementation Update: Package-owned Storybook

The shared component and WidgetRenderer stories have now moved from `web/src` into `packages/rag-evaluation-site/src`, and the package has its own `.storybook` configuration plus `storybook` / `build-storybook` scripts. `web` keeps only app-specific, organism, corpus, workflow, page, retro, and migration parity stories.

The web Storybook also gained a `MockApiProvider` helper and stories for backend-connected views that previously had no direct Storybook coverage. Those stories are deliberately `web`-owned because they exercise app containers and RTK Query integration, not package-level reusable components.

## Risks

- **Storybook story IDs changed** for moved shared stories because they are now discovered from the package Storybook build. Use package Storybook for canonical reusable component review and web Storybook for app/container review.
- **CSS duplication can hide bugs** because package components import `styles.css`; ensure web still imports the correct theme/tokens once local components are removed.
- **Package source imports may pull CSS twice**; test visual output and bundle behavior.
- **Deleting local components too early** can break relative story imports or css-entry reference pages.

## Decision Records

### DR-1: Package becomes canonical for reusable components

Status: proposed

Decision: `packages/rag-evaluation-site` should own reusable foundation/atoms/layout/shared molecules/widgets. `web` should consume them via workspace dependency.

Rationale: Current duplication is already a copied subset; package versions include newer WidgetRenderer fixes; future context-window/Widget DSL work needs a stable reusable target.

### DR-2: Move shared Storybook ownership into the package

Status: accepted

Decision: `packages/rag-evaluation-site` owns stories for exported reusable atoms, foundation primitives, layout primitives, shared molecules, and `WidgetRenderer`. `web` owns app/container stories and the temporary package parity comparison story.

Rationale: The package is now the canonical reusable layer, so its Storybook must be reviewable without relying on web-local duplicated component files. `web` still needs Storybook for backend-connected pages, corpus/workflow organisms, and migration parity checks.

### DR-3: Do not move backend-connected organisms yet

Status: proposed

Decision: Move only API-independent presentational components initially. Leave backend-connected views/containers in `web`.

Rationale: The package should not depend on the RAG backend store/API surface.
